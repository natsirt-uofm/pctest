02 Problem Statement

Why Automate CWE Classification?

The Challenge

  • NVD adds ~25,000 new CVEs per year — manual triage is unsustainable
  • Each CVE is mapped to a CWE (weakness type) that guides remediation
  • Assigning CWE labels requires security expertise and deep code reading
  • Mislabeled CWEs lead to incorrect tool configuration and missed fixes

Goal

Given a C/C++ function, predict its CWE category automatically.

Impact

  • Automated developer advisories
  • CI/CD security gate labeling
  • Faster vulnerability database enrichment
  • Targeted static analysis rule selection

Task Definition

Input: raw function source code (C/C++)
Output: one of K = 10 CWE class labels
Metric: Accuracy & Macro-F1

06 Baseline Model

TF-IDF + Logistic Regression

Why Character N-grams?

  • Capture syntactic micro-patterns: ptr->, free(, strcpy
  • Robust to variable renaming
  • No tokenizer needed — language-agnostic
  • Fast to fit (<5 min on CPU)

07 CodeBERT Model

CodeBERT Fine-Tuning

Pre-Training (Microsoft, 2020)

  • RoBERTa architecture — 12 layers, 768 hidden, 125M params
  • Pre-trained on CodeSearchNet: 6.4M NL-code pairs (6 languages)
  • Objectives: MLM + RTD (replaced token detection)
  • Bimodal: understands both docstrings and source tokens

Classification Head

RobertaForSequenceClassification └ RobertaModel (frozen or fine-tuned) └ Linear(768, K) # K = num CWE classes └ Softmax Loss: CrossEntropyLoss

Fine-Tune Strategy

  • All parameters trained jointly (no layer freezing)
  • Warmup: 10% of total steps → linear decay to 0
  • Gradient clipping: ℓ2 norm ≤ 1.0
  • Best checkpoint by validation Macro-F1

Strengths & Weaknesses

  • ✓ Contextual, semantic representations
  • ✓ Subword tokenizer handles identifiers
  • ✗ 512-token hard limit
  • ✗ Requires GPU for practical training

10 Discussion

What Do the Results Tell Us?

Why CodeBERT Wins

  • Contextual token representations vs. bag-of-ngrams
  • Pre-training on 6.4M code–NL pairs captures syntax + semantics jointly
  • Subword tokenizer handles identifiers like malloc, pthread_mutex
  • Larger gain on semantically complex CWEs (race conditions, UAF)

Confusion Patterns

  • CWE-119 ↔ CWE-787 ↔ CWE-125 — all memory-boundary violations, structurally similar
  • CWE-20 ↔ CWE-200 — input validation vs. disclosure, thin boundary
  • [FILL IN: your top confusion pair from the confusion matrix]

Limitations

  • Top-10 CWEs only — long-tail unaddressed
  • 512-token truncation drops context for long functions
  • Class imbalance despite balanced weighting
  • C/C++ only; not generalizable to other languages without retraining

Future Work

  • LongFormer / sliding-window for long functions
  • Graph-based (GraphCodeBERT, data flow)
  • Data augmentation for minority CWEs
  • Multi-label classification (co-occurring CWEs)
11 Conclusion

Summary

What We Did

  1. Framed vulnerability type prediction as multi-class classification (K=10 CWEs)
  2. Built a TF-IDF + Logistic Regression baseline on BigVul
  3. Fine-tuned microsoft/codebert-base end-to-end
  4. Evaluated with accuracy, macro-F1, precision, and recall on a held-out test set
  5. Analyzed per-class performance and confusion patterns

Key Findings

Takeaway

Pre-trained code representations transfer effectively to fine-grained CWE classification, making automated vulnerability labeling practical at dataset scale.
